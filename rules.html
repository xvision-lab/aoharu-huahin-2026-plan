運営ルール（内部憲章 / Single Source of Truth）

本ページは あおハル運営メンバー＋KAZ の確認・合意形成用。
対外共有（参加者 / タイ側 / SNS）を前提としない。

MENU: 旅程（Tour） スポット（Spots） クリニック検討（Clinic） 確定スケジュール（Program） 参加者動線（Flow） 運営ルール（Rules）

このページの目的

①「どこに何を書くか」 ②「誰が決めるか」 ③「未確定の扱い」 ④「更新ルール」
を一本化し、現場の迷い・揉め・情報分散を先に潰す。

1. 前提（内部専用ルール）

“誰向け” を明確にして、共有ミスを防ぐ
  • 対象：あおハル運営メンバー＋KAZ（内部確認用）
  • 非対象：参加者配布／タイ側共有／SNS告知（別資料にする）
  • 言語：日本語（検討メモを含む）
  • 目的：運営の判断基準と情報整理の“型”を固定する

2. 用語・表記ルール（TBDの説明を含む）

一般の日本人は “略語を知らない前提” で統一する
TBD：To Be Determined（未確定／今後決める項目）
使う理由：空欄や曖昧表現で「見落とされる」のを防ぎ、決める必要がある箇所を目立たせるため。
運用：確定したら TBDを必ず消して具体値に置換する（TBD放置禁止）。
表記 意味 使う場所
TBD 未確定（今後決める） program / flow の「将来確定する枠」
内部メモ 運営検討用の叩き台 index / clinic / rules のみ（対外は禁止）
確定 参加者にも共有できる状態 clinic-program（確定事項のみ）
重要：TBD は参加者配布版・告知文には使わない（別ファイルで “確定版” を作る）。

3. ドキュメント構造（どこに何を書くか）

情報が混ざると「決まってないのに決まった扱い」になるので分離
ページ 役割 書いて良い内容
index.html ツアー旅程（運営叩き台） 全体像／移動／家族側オプション／運営メモ（要検討）
spots.html スポット一覧（補助資料） 観光候補の説明・写真・注意点（体力/年齢配慮）
clinic.html クリニック運営の検討ページ 論点／選択肢／未確定事項／前提条件（日本語の検討メモ）
clinic-program.html 確定プログラム（確定事項のみ） 確定タイムテーブル・担当・配布物・式次第（TBDは“枠”のみ）
clinic-flow.html 参加者動線（集合/移動/解散） 集合場所・時刻・送迎台数・点呼・帰路など実務導線
rules.html 運営ルール（本ページ） 判断基準／運営体制／更新・共有ルール／用語統一
運用原則：「未確定の議論」は clinic.html に戻す。「確定した内容」は program / flow に反映して整合させる。

4. 運営体制（誰が何を担当するか）

“迷ったら誰に聞くか” を固定して、現場判断を速くする
役割 担当 責任範囲（何を決めるか）
総合リーダー 青やん 全体方針・優先順位・最終方向性の確認（大枠の意思決定）
サッカークリニック・リーダー 梅ちゃん、KAZ 8/15–16 クリニック内容・運営判断（クラス設計、必要人員、当日運用）
オフ会ツアー・リーダー ゆーりちゃん、RIE ツアー行程・家族側配慮・観光判断（残留組の満足度設計）
全体コーディネーター KAZ 全体整合（旅程×動線×現地調整のハブ）／タイ側とのすり合わせ窓口
注意：この体制表は “内部運営のための” 明確化。参加者向けの表現は別資料で簡略化する。

5. 意思決定ルール（検討フェーズの原則と線引き）

本セクションは「実行時の優先順位」を定めるものではなく、 検討段階で思考と議論を混線させないためのルールを定める。
前提： 現在は企画・検討フェーズであり、現場実行時の優先順位や即断ルールを 先に固定する段階ではない。

5-1. 検討の三本柱（並列・同時進行）

本プロジェクトは、以下の三つの検討カテゴリを 上下関係なく、すべて同等に扱う。

  • ① あおハル・オフ会ツアー全体の検討
    旅程全体の構造、滞在配分、参加者構成（家族・非参加者含む）、 全体像としてのバランス設計。
  • ② ツアー（旅行・家族・動線）専門の検討
    移動、集合・解散、観光、自由時間、残留組の過ごし方、 子ども引率や安全配慮に関する設計。
  • ③ サッカークリニック専門の検討
    日時・場所、クラス設計、コーチ数、通訳体制、 参加者管理、当日運営に関する検討。
これら三本柱は優先順位の関係ではなく、相互依存関係にある。 一方の最適化が、他方の前提条件を崩す可能性があることを常に意識する。

5-2. 検討時の線引きルール（混同防止）

  • 各議論は、「どのカテゴリの話か」を明示したうえで行う。
  • ある判断が他カテゴリに影響する場合は、 それを「影響」として整理し、論点を混ぜない。
  • クリニック視点のみ、またはツアー視点のみでの最適化を、 全体最適と誤認しない。

5-3. TBD（未確定事項）の位置づけ

  • TBD は「決まっていないから保留」ではなく、 検討が必要な論点を可視化するための印とする。
  • 検討フェーズでは TBD の存在自体を問題視しない。
  • 実行フェーズに入る前に、TBD は必ず整理・確定・削除する。
注意： 実行段階（現場での優先順位・即断ルール）は、 本セクションでは扱わない。必要になった時点で、 別途整理・明文化する。

6. タイ側（Coach Gee）との運用ルール

日本語の検討資料をそのまま渡さない（別チャンネルで整理）
方針：Coach Gee とは 別チャンネルでKAZが整理してすり合わせする。
日本語の検討メモ（clinic.html / rules.html 等）をそのまま共有しない。
  • タイ側へ渡すのは「確定事項」中心（時間・場所・人数・必要物）。
  • 未確定を渡す場合は「決めるべき問い」として要点化し、英語化して伝える。
  • 通訳者・補助人員の依頼は、要件（1st/2nd/補助）を明確にして依頼する。
